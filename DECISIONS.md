# Locked Decisions — إلكترونك-سيتي Refactor

This file records the architectural and implementation decisions for the
إلكترونك-سيتي refactor. Every choice here is **deliberate**; deviations
should be discussed in a PR and updated here.

---

## D1 — Persistence Library: Room ✅

We stay on **Room** (Google, KSP codegen). Rationale:
- Already in production use, 4 entities + 4 DAOs working
- Android-only product, no iOS KMP target
- Library switch during a schema refactor is a "two big changes" anti-pattern
- Re-evaluate only if a shared `:data` module or KMP becomes a roadmap item

## D2 — Tab Numbering: forward, locale-driven layout direction ✅

The original D2 spec was to **keep reverse indexing** (Home=4, Account=0)
during the refactor for zero behavior change. That constraint is **now
obsolete** (see D8.20): the enum has been flipped to forward numbering
(`HOME(0), PRODUCTS(1), DISCOUNTS(2), FAVORITES(3), ACCOUNT(4)`) because
layout direction is now derived from the active locale (D8.19) rather than
hardcoded RTL — reverse indexing no longer carries RTL information.

The full i18n plan it described is now executed:

- ✅ `values-ar/strings.xml` (existing Arabic resources)
- ✅ `values-en/strings.xml` (D8.18 — full English translation)
- ✅ `Locale` driven `LocalLayoutDirection` (D8.19 — `LocaleManager.wrapFromPrefs`)
- ✅ `enum class HomeTab(val index: Int)` is the source of truth
- ✅ `MainActivity` picks layout direction from the active locale (D8.19 — note: we did **not** use `AppCompatDelegate.setApplicationLocales(...)`; see D8.19 for the trade-off and the `LocaleManager` approach)

## D3 — Font Family: Google Fonts with layered fallback 🔤

We use **Google Fonts** via `androidx.compose.ui:ui-text-google-fonts`, with
a layered fallback chain so the app renders correctly on first run with no
network:

1. **First choice:** Google Fonts (IBM Plex Sans Arabic + IBM Plex Sans)
2. **Second choice:** System font (FontFamily.Default / FontFamily.SansSerif)
3. **Third choice (deferred):** Bundled `.ttf` assets in `res/font/`

The provider is `GoogleFont.Provider` with `com.google.android.gms.fonts`
authority; Compose caches the font after first successful fetch, so the
fallback only triggers once per device.

**Configuration strategy:**

```kotlin
val provider = GoogleFont.Provider(
    providerAuthority = "com.google.android.gms.fonts",
    providerPackage = "com.google.android.gms",
    certificates = R.array.com_google_android_gms_fonts_certs
)
val DisplayFont = FontFamily(
    Font(GoogleFont("IBM Plex Sans Arabic"), provider, FontWeight.Bold),
    Font(GoogleFont("IBM Plex Sans Arabic"), provider, FontWeight.SemiBold)
)
val BodyFont = FontFamily(
    Font(GoogleFont("IBM Plex Sans Arabic"), provider, FontWeight.Normal)
)
```

Compose's `Font` API takes a `fontProvider` argument; if the fetch fails it
falls back to the next item in the `FontFamily` chain (which we will set to
the system font), giving a graceful offline first-run.

## D4 — Static Analysis: detekt ✅

We adopt **detekt** (1.23.x) with `config/detekt.yml` and a baseline.
Rationale:
- Catches real code smells (LongMethod, TooManyFunctions, MagicNumber) that
  the review identified
- Built-in rule sets: `style`, `complexity`, `exceptions`, `performance`,
  `naming`, `potential-bugs`
- Kotlin-first, easy CI integration
- Customizable per project

ktlint was rejected because it only does style enforcement, not bug detection.

## D7 — ViewModel Scoping: 6 Activity-scoped focused VMs ✅

We split the God `ECommerceViewModel` (426 lines, 8 responsibilities) into
**6 focused ViewModels**, all scoped to `MainActivity`'s
`viewModelStoreOwner`:

1. `ThemeViewModel` — `themeMode`, `setThemeMode(...)`
2. `UserProfileViewModel` — user profile + online customers
3. `ShopViewModel` — products, filters, `ProductsUiState`, detail selection
4. `CartViewModel` — cart items, total, count
5. `OrderViewModel` — orders, COD checkout, offline sync
6. `NavigationViewModel` — tab history, back stack

A small `AppViewModels` data class bundles them so screens can request
exactly the subset they need. If a future `NavHost` adds a real checkout
destination, that one VM will be scoped to its own `NavBackStackEntry`.

- **D8.18** `values-en/strings.xml` (Future i18n Backlog, D2 step 1) — full English translation of all 241 keys from `values/strings.xml`. Uses **semantic translation** rather than literal: e.g. `cd_close = "Close"` even though the Arabic value `"عودة"` literally means "return" (pre-existing typo in `values/strings.xml`, preserved to avoid behavior change; deferred fix tracked in `plan.md` → Future i18n Backlog). XML well-formed; key set matches `values/strings.xml` exactly (verified via PowerShell `Compare-Object`). - **D8.19** In-app language switcher (Future i18n Backlog, D2 step 2; revises the "AppCompatDelegate" line above) — `LocaleManager` (new) wraps the base context of `MainActivity` with the persisted BCP-47 tag via `Context.createConfigurationContext`; `MainActivity.setAppLocale(tag)` persists the choice and calls `recreate()`. `AccountScreen` shows a "Language" card mirroring the theme card, with two `LanguageOptionButton`s ("العربية" / "English") that highlight the active tag. **No new dependency**: deliberately does **not** use `AppCompatDelegate.setApplicationLocales(...)` because that would require adding `androidx.appcompat:appcompat` (~700KB) and switching `MainActivity` from `ComponentActivity` to `AppCompatActivity`. The trade-off is that the per-app locale does not surface in the Android 13+ system per-app-locale settings (in-app switcher only). Acceptable for now; revisit if a system-settings entry becomes a hard requirement. The hardcoded `CompositionLocalProvider(LocalLayoutDirection provides LayoutDirection.Rtl)` was **dropped** from `MainActivity`; layout direction is now derived from the active locale (Arabic = RTL, English = LTR). Default persisted tag is `"ar"`, matching the Arabic-first `values/strings.xml` default and the original hardcoded RTL behavior — Arabic-system users see no change on first launch. **Important invariant for reviewers**: switching language triggers a normal `Activity.recreate()`, so `rememberAppViewModels` re-derives the activity-scoped VMs from the factory but does **not** re-fetch server data (`ShopViewModel` survives via the activity's `ViewModelStore`).
- **D8.20** `HomeTab` forward numbering (Future i18n Backlog) — flipped the enum from the historical reverse numbering (`HOME(4), ACCOUNT(0)`, where `index` doubled as "visual position from the left in RTL") to forward numbering (`HOME(0), ACCOUNT(4)`, matching `entries[i]` declaration order and the LTR convention). The visual order of the `NavigationBar { ... }` items in `MainScreen.kt` is **unchanged** (Home first, Account last in source order), so the rendered tab order is still correct in both Arabic (RTL: Home on the right, Account on the left) and English (LTR: Home on the left, Account on the right) — only the `index` field values change. The historical `// (index 4)` / `// (index 0)` comments next to the Home/Account `NavigationBarItem`s were updated to `// (index 0, D8.20)` / `// (index 4, D8.20)`. `HomeTabTest.kt` added (4 tests: forward indices, declaration order, `fromIndex` round-trip, unknown-index fallback to `HOME`) to lock the contract against future regressions — a silent index shift would route the user to the wrong screen in `when (HomeTab.fromIndex(selectedTab))` dispatch.
- **D8.21** Locale-aware VM string resolution (Future i18n Backlog) — the `Application` instance is created once per process and its base context is never re-wrapped, so `getApplication<Application>().getString(R.string.xxx)` would **always** resolve to the **default** locale (Arabic) — even after the user picks English via D8.19's in-app switcher. To fix this, `LocaleResources` (new) is an `AtomicReference<Context>` singleton that `MainActivity.attachBaseContext` publishes its wrapped base context to (runs before `super.onCreate`, so the ref is set before any VM is constructed). All 16 `getApplication<Application>().getString(R.string.xxx)` call sites in `CartViewModel` (1), `OrderViewModel` (9), `ShopViewModel` (4), and `UserProfileViewModel` (2) now use `LocaleResources.getString(R.string.xxx)` (with a `formatArgs` overload for `R.string.msg_orders_resynced`'s `%1$d`). The atomic ref is updated on every `attachBaseContext` (i.e. on every activity recreate, including locale change), so VM coroutines that publish after a language switch resolve the new locale on the very next read. **`LocaleResourcesTest.kt`** locks the contract with 3 Robolectric tests: empty-string fallback before `set`, Arabic vs English resolution via `LocaleManager.wrap(base, "ar")` / `wrap(base, "en")`, and `%1$d` format-args. Why not `LocalContext.current.getString(...)` (the original Future i18n Backlog phrasing)? `LocalContext` is a Compose composition local — VMs run inside `AndroidViewModel`s, not in the composable scope, so they cannot see it. The singleton bridge preserves the D8.10 invariant that VMs stay Context-free at construction time.

- **D8.22** `typealias Customer = UserProfile` (Phase 7B-2) — `AdminCustomersBody(customers: List<Customer>, …)` at `AccountScreen.kt:1994` was a real compile break: the wildcard import `com.example.data.model.*` (line 30) does not bring in any `Customer` class. The minimum-disruption fix is a `typealias` in `data/model/Customer.kt` (`typealias Customer = UserProfile`); zero call-site changes, and the typealias is forward-compatible — if a future `Customer` model with extra fields is ever introduced, the typealias gets replaced by a real `data class Customer(...)` and the compiler flags the call sites that need to migrate. The call site at `AccountScreen.kt:1439-1444` already passes `vms.userProfile.onlineCustomers: StateFlow<List<UserProfile>>`; the typealias is transparent there. Considered alternatives: (a) a full `data class Customer(...)` duplicating the fields `onlineCustomers` exposes — rejected because it introduces two sources of truth (drift risk), (b) renaming `UserProfile` → `Customer` everywhere — rejected because the blast radius extends into the auth/profile surfaces that are outside the admin audit's scope.

- **D8.23** `AdminAuthViewModel` (Phase 7B-2) — the admin gate at `AccountScreen.kt:65` (`var isAdminAuthenticated by remember { mutableStateOf(false) }`) plus the hardcoded `equals("admin123")` check at line 896 were both a security risk (password literal in source) and a UX bug (a locale switch — `Activity.recreate()` — would re-execute the `remember` initializer, but D8.19's `LocaleManager.wrap` is invoked at `attachBaseContext` *before* the `remember` runs, so the `mutableStateOf(false)` reset is fine; the actual bug is that the `remember` scope survives `recreate()` for the admin gate specifically, which masks the security gap). New `ui/viewmodel/AdminAuthViewModel.kt` (an `AndroidViewModel`, Activity-scoped per D7) owns `isAuthenticated: StateFlow<Boolean>`, `failedAttempts: StateFlow<Int>`, `lockoutUntilMs: StateFlow<Long?>`, `lastInteractionMs: StateFlow<Long>`. Methods: `signInWithFirebase(password: String)` calls into a new `RemoteDatabaseService.requireAdmin(): Result<Unit>` (D9.3 stub returns `Result.success(Unit)` until the real Firebase project lands — see the D9.3 note about how the admin gate is *untrusted* until the stub becomes a real claim check), increments `failedAttempts` on failure, and sets `lockoutUntilMs = now + 60_000` after 5 failed attempts within 60 s. `recordActivity()` is called on every `onClick` in the admin area; a 30-s polling coroutine in the VM locks the session if `now - lastInteractionMs > 5 * 60_000`. An explicit "Lock admin session" `IconButton` in the admin tab row calls `lockNow()` and closes the sheet. Because the VM is Activity-scoped, it survives `LocaleManager`-triggered `recreate()` — so the auth state is correctly preserved across locale switches (the gate auto-redirects to admin tabs on a successful `isAuthenticated` observation, the new locale takes effect, the admin stays logged in). The 4 `AdminAuthViewModelTest.kt` cases (Robolectric) lock the contract: successful sign-in, 5/60s lockout, locked-state rejection even with correct password, 5-min idle timer.

- **D8.24** `Product.archivedAt: Long?` + Room `Migration(2, 3)` (Phase 7B-2) — soft-archive for the admin delete action. The `OrderItem` entity has a `ForeignKey(entity = Product::class, …, onDelete = ForeignKey.RESTRICT)` at `Product.kt:71-76`, so a hard delete on a product that's been ordered fails at the DB level — that fragility was the audit's H2 finding. Soft-archive adds a nullable `archivedAt: Long? = null` column to `Product`; storefront queries (`getAllProductsFlow`, `getFeaturedProductsFlow`, `getDiscountedProductsFlow`, `getProductsByCategoryFlow`) get a `WHERE archivedAt IS NULL` filter; a new `getArchivedProductsFlow` powers a 4th "Archived" admin tab with restore capability; archive/unarchive go through `archiveProduct(id, archivedAt)` and `unarchiveProduct(id)` DAO methods. The schema bump from v2 → v3 is the first real migration in the project (the v1 → v2 bump in Phase 1.1 relied on `fallbackToDestructiveMigration()`). The new `data/local/Migrations.kt` declares `MIGRATION_2_3 = object : Migration(2, 3) { override fun migrate(db) { db.execSQL("ALTER TABLE products ADD COLUMN archivedAt INTEGER") } }` and `AppDatabase.getDatabase` chains `.addMigrations(MIGRATION_2_3)` *before* `.fallbackToDestructiveMigration()` (the destructive fallback stays as a safety net for v0/v1 users migrating from a pre-MIGRATION_2_3 state, but the real migration handles the v2 → v3 case deterministically — no data loss for users on the v2 schema). `MigrationTest.kt` (Robolectric) opens the DB at v2 with a seeded product, runs `MIGRATION_2_3.migrate(db)`, and asserts the v3 read sees the same row with `archivedAt = null` — the test catches future schema-drift regressions where a contributor might break the SQL and ship a destructive fallback to production users. The M3 decision (wire the discount fields) does **not** add columns: `Product.isDiscounted` and `Product.originalPrice` already exist (`Product.kt:24, 20`) — the M3 issue is purely a 1-line wire-through at the admin upload call site (`AccountScreen.kt:1383-1393`), not a schema change.

- **D8.25** `OutOfStockException` + atomic stock re-read (Phase 7B-2 H3) — the admin add-to-cart / place-order paths had a TOCTOU race and an unlimited cart-add path (no re-check against current stock). The fix splits the responsibility: `CartRepository.addToCart` re-reads `Product.stock` from the DAO and throws `OutOfStockException` if the request exceeds the current `availableStock`; `OrderRepository.placeCODOrder` re-reads inside `database.withTransaction`, decrements each item's stock with a guarded `UPDATE … WHERE stock >= :quantity` SQL (the 0-row return is treated as a stock mismatch), and the entire transaction rolls back if any single item is OOS — so a partial decrement never persists. `OutOfStockException` is a `RuntimeException` (not checked) because Room's `withTransaction` doesn't propagate checked exceptions cleanly. `CartViewModel.addToCart` / `OrderViewModel.submitCODCheckout` catch the exception and publish a `MessageBus` event (D8.21 invariant): `msg_cart_out_of_stock` when `availableStock == 0`, `msg_cart_out_of_stock_with_count` with the available count otherwise, `msg_order_out_of_stock` on the order path (different mental model — the user clicked "Place order", copy is order-level). `ProductDetailScreen` Add-to-Cart button gains `enabled = product.stock > 0`; the existing `detail_stock_out` chip at lines 263–275 was already there pre-P5. The headline Robolectric test (`placeOrder_rollsBackPartialDecrement_onStockFailure` in `OrderRepositoryStockTest`) seeds a cart with 2 items, sets item 2's `stock` below the requested quantity, and asserts that item 1's stock is **untouched** after the failure — locking the all-or-nothing contract. The user-facing `Sold out` badge on storefront cards (T5.8) is deferred because there is no shared `ProductCard` component; the storefront renders cards inline in `HomeScreen.kt` / `ProductsScreen.kt` / `DiscountsScreen.kt`, and adding the badge to 3 sites independently is a separate UI pass. The `ProductRepository.placeCODOrder` order of operations (insert order → insert order items → decrement stock, all in one transaction) is the only way to get the atomicity we need without an external lock — a `synchronized` block at the VM layer would be wrong because the mutation must survive a process death and resume on a fresh process.

## D9 — Post-i18n enhancements (Phase 9)

Four new work items that were deferred from Phase 6 (per D5) and the
i18n backlog. Each is self-contained except D9.4 which depends on D9.3.

> **Sequencing recommendation**: do them in the order D9.1 → D9.2 →
> D9.3 → D9.4. Pure-UI first (D9.1, D9.2), backend-heavy last
> (D9.3, D9.4). D9.4 is meaningless without D9.3.

- **D9.1** `WindowSizeClass` adaptive layouts — `MainActivity.onCreate` calls `calculateWindowSizeClass(this)` and publishes the result via a new `LocalWindowSizeClass` composition local (`staticCompositionLocalOf`, default = `WindowSizeClass.calculateFromSize(DpSize(360.dp, 640.dp))` for previews/tests). Chose `staticCompositionLocalOf` over `compositionLocalOf` because the value is read by a handful of composables and never changes during the activity lifetime — Compose skips invalidation tracking. Replaced the hardcoded `LazyVerticalGrid(GridCells.Fixed(2), ...)` in `ProductsScreen.kt`, `FavoritesScreen.kt`, and `DiscountsScreen.kt` with `GridCells.Adaptive(minSize = …)` where the minSize scales with the width class (180dp Compact / 220dp Medium / 260dp Expanded). Preferred `GridCells.Adaptive` over a `when`-on-`widthSizeClass` because the column count then derives from the actual width, not from a guess — a 700dp wide surface gets 3 columns automatically, not 2. `HomeScreen` carousels extracted into a private `ProductsCarousel` helper that picks `LazyRow` (with `width=175.dp` at Compact / `width=220.dp` at Medium) or `LazyVerticalGrid(GridCells.Fixed(2))` at Expanded. `ProductColumnItem` gained a `width: Dp? = 175.dp` parameter so the same composable works in both row and grid contexts. **`AccountScreen` admin tabs branch** at `Expanded` width: Inventory (tab 1) and Customers (tab 2) split into a `DraggableSplitRow` with a draggable 8dp splitter and a 24dp grip pill. Two new private composables — `AdminInventoryBody` and `AdminCustomersBody` — extracted so the same body renders in the left and right panel without duplicating ~250 lines. Add/Edit Product form (tab 0) stays full-width because it's too vertical to fit alongside another panel. Compact / Medium widths keep the original single-column if/else if dispatch untouched. Drag ratio is coerced into `[0.2, 0.8]` with a secondary 200dp-per-panel floor so neither side can collapse. The `RemoteDatabaseService` contract is unchanged — this is a pure UI pass, no VM or repository surgery. New dep: `androidx.compose.material3:material3-window-size-class` (BOM-managed, no explicit version); the annotation is `@ExperimentalMaterial3WindowSizeClassApi` and the call site (`MainActivity.onCreate`) is opted in implicitly because the call is inside a non-preview code path.

- **D9.2** Empty-state illustrations slot — New `ui/components/EmptyState.kt` with signature `@Composable fun EmptyState(illustration: Painter?, title: String, subtitle: String? = null, action: (@Composable () -> Unit)? = null)`. Illustration is `contentDescription = null` (decorative — let the title do the screen-reader work), 120dp tall, centered. **Audit found 7 distinct sites** (the plan originally said 11, but consolidating the 3 admin panels in `AccountScreen` + 4 user-facing screens gives 7): search empty in `ProductsScreen`, discounts empty in `DiscountsScreen`, wishlist empty in `FavoritesScreen` (was using `Icons.Filled.Favorite` 64dp — replaced by the new `empty_wishlist_art` drawable), cart empty in `CartScreen` (the only site that needs the optional `action` slot for the "Browse store offerings" button), plus inventory/customers/orders empty states in `AccountScreen` (these pass `illustration = null` because the admin panels are small-text 11sp/13sp and don't need a hero illustration; the orders empty state gets `fillMaxWidth().height(220.dp)` to give the sheet breathing room). The 4 user-facing sites previously had subtle font-size and alpha drift (13sp vs 14sp vs 16sp, alpha 0.7 vs 0.8) that centralizing through `EmptyState` makes uniform. **Ship without a designer**: 4 placeholder vector drawables (`empty_search_art.xml`, `empty_discounts_art.xml`, `empty_wishlist_art.xml`, `empty_cart_art.xml`) are minimal Material outline shapes (24×24 viewport, `fillColor="#FF000000"` so `ColorFilter.tint(outline@50%)` in `EmptyState` can recolor them). Each drawable ships with an XML comment naming the artwork to drop in and the recommended canvas size (240×240dp, 1.5x density). No `cd_*` strings needed for v1 because the illustrations are decorative; add them only if/when an illustration becomes semantically meaningful.

- **D9.3** Real Firebase Auth + Firestore — **External prerequisite**: user provides a Firebase project + a dev/QA `google-services.json` (kept out of git per line 22 of `.gitignore`; production gets its own via CI secret). Add `com.google.gms:google-services` plugin to `libs.versions.toml` (version `4.4.2`) + root + app build files. **Plugin application is conditional** in `app/build.gradle.kts` via `if (file("google-services.json").exists())` so a fresh clone (no JSON) still builds; drop the JSON in to auto-apply.
